# 玩法工程规范

> 本文档供生成玩法的 AI 读取。生成任何玩法 HTML 时，必须遵守以下规范。
> 同目录配套：`func.md`（埋点接口正本）、`report.js`（埋点标准实现）、`demo骨架/`（结构参照）。

---

## 一、架构

1. **默认单文件**：一个玩法 = 一个 `index.html`，HTML/CSS/JS/题库全部内嵌。图片、音频、视频等二进制资源放 `assets/`。
2. **何时才拆多文件**（满足任一才考虑，否则一律单文件）：
   - 同一玩法要复用给 ≥3 个 bot；
   - `index.html` 超过约 2000 行；
   - 玩法需要「个性化离线预生成」（每人题目不同，数据必须与模板分离）。
3. **强制配置区**：无论单文件还是多文件，所有「换 bot / 换玩法会变的东西」必须集中在文件顶部一个配置区，包括但不限于：
   - `REPORT_CONFIG`（api_url、sceneName、bot_name、game_name）
   - 题库 / 内容数组
   - 主题色、头像/图片路径、文案
   配置区之外的逻辑代码不得硬编码这些值。

---

## 二、埋点（强制）

1. **必须复用 `report.js` 的实现**，不要自行重写上报逻辑。单文件玩法把其内容内嵌进 `<script>`；多文件玩法用 `<script src="report.js"></script>`。
2. **接口与字段严格遵循 `func.md`**，不得增删字段、不得改字段名。
3. **必报的标准生命周期事件**：
   - `pv`：页面加载完成，报一次。
   - `exp`：页面卸载/切后台时报，**必须带 `eventTime`（停留时长）**。
   - 必须同时监听 `visibilitychange`（hidden 时）和 `pagehide`，且卸载上报用 `sendBeacon` 或 `fetch + keepalive`。**不得只用普通 fetch，否则数据会丢。**
4. **业务点击事件**命名为 `{用途}_clk`，小写英文+下划线，如 `start_clk`、`submit_clk`、`share_clk`、`retry_clk`。不得用中文或特殊字符。
5. **一局一个 `contentId`**：进入页面生成；「再来一次」开新局时必须重新生成 `contentId` 并重置 `pageEnterTime`。

---

## 三、回传给对话模型的 content（摘要）

1. 在「提交完成 / 进入聊天 / 中途离开」等需要让 bot 知情的节点，通过 `content` 字段回传一段摘要。
2. content 规范：
   - **纯文本**，不是 JSON；
   - **最大 10000 字符**；
   - **不含 HTML/脚本**（后台会做 XSS 过滤）；
   - 用**规则拼接**生成（参照 demo 的 `buildSummaryText()`），**不要调用 LLM**。
3. 摘要是**自然语言叙述**，让 bot 读完能直接开启个性化对话（包含：时间、玩法名、完成情况、用户的具体选择）。

---

## 四、前端与兼容

1. 纯前端，无构建步骤，无框架依赖（不引入 React/Vue 等）。可用原生 ES6+。
2. **移动端优先**：`max-width` 约 520px 居中；加 viewport meta 禁止缩放。
3. 外部依赖尽量内联；必须用 CDN 时只用稳定公共 CDN。
4. 所有上报与网络请求**异步非阻塞**，失败用 `.catch()` 兜住，不得阻断玩法主流程。
5. 资源路径用相对路径（`assets/xxx`、`images/xxx`），不要写死本地绝对路径。

---

## 五、目录约定

```
玩法名/
├── index.html          # 主入口（单文件时所有逻辑在此）
├── report.js           # 仅多文件玩法需要；单文件玩法内嵌后可不放
└── assets/             # 图片、音频、视频等资源
```

---

## 六、不要做的事

- 不要把 `userToken` 等 URL 参数单独解析后再拼装；`urlParams` 整体透传。
- 不要在 content 里塞结构化 JSON 让 bot 解析；用自然语言。
- 不要因为「规范化」就擅自把单文件拆成多文件，除非命中第一节第 2 条。
- 不要修改 `func.md` 中定义的接口地址、字段名、事件命名规则。
